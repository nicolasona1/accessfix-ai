# AccessFix AI

Hackathon MVP for hearing and fixing web accessibility barriers.

## What This Project Does

AccessFix AI lets developers paste HTML or analyze a URL, detect common accessibility issues, hear a screen-reader-style simulation of the page, generate reviewable fixes, and compare the before/after experience.

The project was built as a hackathon prototype for accessibility-focused developer tooling. It is not a full WCAG certification product, but it demonstrates a working end-to-end accessibility repair workflow.

## Technologies Used

- React and Vite for the frontend
- Custom CSS for the dark developer-tool interface
- Framer Motion for UI animation
- Lucide React for icons
- Browser Web Speech API for spoken screen-reader-style simulation
- DOMParser for client-side HTML parsing
- FastAPI for the backend API
- BeautifulSoup4 for backend HTML analysis
- Requests for URL fetching
- Pydantic for API schemas
- OpenAI SDK for optional model-powered fixes
- Ollama for optional local model-powered fixes
- Local deterministic accessibility repair rules as a reliable fallback

## AI Usage & Attribution

ChatGPT was used as a development assistant during this project. It helped with:

- Providing feedback on how to approach and structure the idea
- Debugging and troubleshooting frontend/backend issues
- Refining the demo flow and presentation strategy
- Assisting with parts of the frontend UI design and code organization
- Helping document the project clearly for hackathon review

The application itself can also use AI/model tools at runtime:

- OpenAI can be used by the backend `/api/fix` route when `OPENAI_API_KEY` is configured and has quota.
- Ollama can be used as an optional local model provider when running on `localhost:11434`.
- If no model provider is available, the app uses local deterministic accessibility repair rules so the demo remains reliable.

All generated accessibility suggestions are intended to be reviewable by a developer before publishing. Image alt text suggestions are marked with `data-ai-suggested="true"` when generated by the repair flow.

Core product direction, final implementation decisions, and hackathon presentation choices were reviewed and selected by the project creator.

## Run The Frontend

```bash
cd frontend
npm install
npm run dev
```

Open `http://localhost:5173`. To scrape a live page, start the backend too, choose `Enter URL`, paste a URL, and analyze. The backend fetches one HTML page and returns the markup to the frontend for issue display and screen-reader simulation.

## Run The Backend

```bash
cd backend
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn main:app --reload --port 8000
```

The backend provides `/api/analyze` for URL scraping and BeautifulSoup accessibility checks, plus `/api/fix` for model-powered suggestions. It tries OpenAI first, then optional local Ollama, then local deterministic fixes.

## Enable Real AI Fixes

Create `backend/.env`:

```bash
OPENAI_API_KEY=sk-your-key-here
```

Restart the backend after adding the key. The frontend `Fix with AI` button calls `/api/fix` once per detected issue. If the key is missing, out of quota, or the API call fails, the app can use Ollama or reviewable local fixes so the presentation still works.

## Optional Local AI With Ollama

Install Ollama and run a local model:

```bash
ollama run llama3.2
```

Optional `backend/.env` settings:

```bash
OLLAMA_BASE_URL=http://localhost:11434
OLLAMA_MODEL=llama3.2
```

The `/api/fix` provider order is OpenAI, Ollama, then local accessibility repair rules.

## Demo Script

1. Start both frontend and backend.
2. Enter a real URL or load the backup demo.
3. Show the score and issue list generated from fetched HTML.
4. Play the before simulation.
5. Apply fixes and show the flagged AI suggestions.
6. Play the after simulation and compare the improved score.
